Blind "detect any of 99 languages" turned out to be unreliable in this browser environment — every unconstrained pass Whisper offers can silently mistranslate instead of transcribing, with no dependable way from here to tell when it has. What does work reliably: naming the language explicitly. If your recording switches between two languages, set both here — each ~15s window is tried against both explicitly and whichever result actually matches its target language's words or script is kept, roughly doubling processing time in exchange for actually being trustworthy.

    How this stays local-first: the model weights (~250 MB, one-time) are
    fetched from Hugging Face's CDN the first time you load the model, then cached by the browser
    for offline reuse. From that point on — and for every recording you make — nothing leaves this
    tab: audio is read, chunked, and transcribed in-memory on your device.


    Known limits of this prototype: recordings over 15s are split into 15s windows
    with 3s overlap, so the model's 30-second context limit never truncates a long recording. Getting
    reliable language handling took several rounds of direct testing before landing here, worth being
    upfront about: this pipeline never exposes a detected-language field through its API at all, and
    forcing task: "transcribe" without also pinning down an explicit language turned out
    to be unreliable on its own — it would sometimes translate to English silently, with no way from
    the outside to tell when it had, confirmed by a "sample" pass coming back as a fully fluent
    English translation of what was actually French. The only combination confirmed reliable, twice
    over, is language and task forced together explicitly. So this prototype no longer attempts blind
    auto-detection across all 99 languages — that's a real capability gap worth knowing about, not
    a hidden one. Instead, "Also expect" lets you name a second language a recording is expected to
    switch into; when set, every ~15s window is transcribed once per candidate language, explicitly,
    and whichever result actually matches its own target language — by Unicode script for languages
    like Tamil and Hindi, by stopword frequency for Latin-script languages like French, English, and
    Tagalog — is kept. This means hybrid mode always costs twice the compute of single-language mode,
    since both candidates genuinely run every window regardless of which one turns out to be right.
    It also can't recover a language from Whisper's own hallucinated, phonetically-transliterated
    output (seen once in testing, where a Tamil section came out as repeated nonsense Latin-letter
    syllables) — no scoring scheme fixes text that was never real to begin with. A generation length
    cap is applied per chunk for that same hallucination case, since the repetition penalty alone
    didn't fully stop it from looping a single word hundreds of times — the cap won't prevent that
    failure mode outright, but it bounds how much damage one bad chunk can do to the rest of the
    transcript. A small silence pad is added to the true end of longer recordings, capped to whatever
    headroom is left in the last 15s window so it can never spill into creating an extra, near-silent
    final chunk — an earlier version of this padding used a fixed amount and occasionally did exactly
    that, which demoted the chunk holding the real final words out of "last chunk" status and got its
    tail trimmed as if it were a middle chunk. When a language switch is detected, it's marked inline
    as ⟨→ Language⟩. None of this catches a switch that happens faster than about 15
    seconds, including mid-sentence code-switching, or a third language beyond the two candidates
    picked — those are hard limits of this approach, not settings to tune away. Shorter windows also
    make Whisper's known repetition/hallucination failure mode — getting stuck looping a phrase,
    usually on a quiet stretch — somewhat more likely, since a short pause now makes up more of a 15s
    window than it would have of a 30s one; a repetition penalty and a no-repeat-ngram constraint are
    applied during generation to push back on it, though this is a mitigation rather than a
    guarantee. There's no speaker diarization here — that's a separate stage in the fuller
    blueprint. Performance depends on your device; a laptop with WebGPU support will be noticeably
    faster than CPU-only WASM, and very long recordings will simply take longer rather than failing.
